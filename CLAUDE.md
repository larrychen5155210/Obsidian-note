# 筆記寫作與架構規則 (Notebook Structure & Writing Rules)

## 1. 標籤 (Tags) 規範
- 所有的 Obsidian 標籤 (Tags) 均必須使用**全小寫**。
- 若標籤含有空格或特殊字元，請以連字號 `-` 或底線 `_` 替換以符合格式（例如：`serv-u-ftp-server`、`bash_history`、`writable-etc-passwd`）。
- **避免重複與相似合併**：在整理 writeup 寫成筆記並新增 tags 時，必須先讀取整個筆記庫原有的 tags list，評估是否有相似的 tags。應優先重用既有 tags，不要直接新增新 tags，以免 tags 太過雜亂。
- **流程規範**：當使用者提供 writeup 連結要求整理時，代理應先自行判斷可以加入哪些 tags，並在正式寫入這些 tags 前，必須先詢問使用者的意見以獲得確認。

## 2. 指令與手法筆記組織
- 每一種滲透測試技術、工具或提權手法（如 `sqli`、`wordpress`、`mysql` , `cron-job`、`suid-serv-u-ftp-server`、`pwnkit`、`writable-etc-passwd` 等）必須建立為**獨立的筆記檔案**，並存放於 `指令_手法/` 資料夾下，避免將多個不同手法混寫於單一檔案中。
- **重要限制：在建立手法筆記前，必須先詢問使用者的意見，確認哪些手法要建立成新筆記並加入指令_手法資料夾。**
- **避免重複建立與優先合併**：在為新的 Writeup 建立手法筆記前，必須先檢視 `指令_手法/` 資料夾內現有的筆記，評估是否有適合合併或歸類在一起的既有筆記。應優先將相似的手法或指令增補、整理至既有筆記中。**在執行合併或增補前，必須先詢問使用者的意見以獲得確認。**
- **範本與主標題規範**：建立新的指令與手法筆記時，必須依據 `Templates/指令手法模板.md` 來撰寫。為了配合雙向連結的精準定位，新手法筆記的 H1 主標題必須以 `💥` 符號開頭（格式如：`# 💥 手法名稱 (小寫標籤)`），使簡介筆記的雙向連結能以此 `💥` 標題作為精準的錨點連結。

## 3. 雙向連結與錨點引用
- **簡介筆記 (Summary.md)** 與 **指令筆記 (Command.md)** 之間必須建立雙向連結：
  - 在簡介筆記中提到特定手法時，使用精準的章節錨點連結至指令筆記（例如：`[[mysql#💥 INTO OUTFILE 寫入 Web Shell|INTO OUTFILE]]`）。
  - 在指令筆記的末尾，必須建立 `# 實戰關聯` 段落並連結回對應的靶機簡介筆記（例如：`[[Stapler]]`）。
- **詳細筆記 (Walkthrough.md)** 應保持純淨，專注於紀錄靶機的真實滲透步驟與終端輸出結果，**不需**在正文中加入指向 `指令_手法/` 資料夾的雙向連結。

## 4. YAML 屬性區塊 (YAML Frontmatter) 規範
- **只有靶機簡介筆記**需要加上 YAML 屬性區塊（YAML Frontmatter）。
- 其他筆記檔案（如 `指令_手法/` 下的手法筆記、詳細 Walkthrough 筆記等）**均不需**加上 YAML 屬性區塊。

## 5. 資料來源 (References) 規範
- 當使用者提供 Writeup 的連結並要求整理成筆記時，必須將這些 Writeup 的連結保留在**靶機簡介筆記**末尾的 `## 🔗 資料來源 (References)` 區段中（格式請參考 `Templates/靶機簡介模板.md`）。
